# TrafficWrapper App

[![CI](https://github.com/TrafficWrapper/app/actions/workflows/ci.yml/badge.svg)](https://github.com/TrafficWrapper/app/actions/workflows/ci.yml)

[Русский](README.ru.md)

Android public client for TrafficWrapper, an open-source self-hosted private
transport platform for small operator deployments and transport-obfuscation
research. The app is not an Android VPN: it exposes a local SOCKS front-end and
routes selected app traffic through signed deployment config and user-space
transports.

The operator owns the deployment keys, worker endpoints, bootstrap payloads, and
update channel. The public app verifies minisign-signed config and update
manifests, pins the APK signing certificate, and can be rebuilt from source for
local audit.

TrafficWrapper is split into three repositories:

- [orchestrator](https://github.com/TrafficWrapper/orchestrator) — the control plane.
- [worker](https://github.com/TrafficWrapper/worker) — REALITY + AmneziaWG data plane nodes.
- [app](https://github.com/TrafficWrapper/app) — this Android client.

The normal workflow is: deploy an orchestrator, enroll workers, build/install
this app, then import a bootstrap payload generated by the orchestrator.

Architecture and threat-model notes live in [ARCHITECTURE.md](ARCHITECTURE.md)
and [THREAT_MODEL.md](THREAT_MODEL.md).

## Troubleshooting

The canonical end-to-end troubleshooting guide lives in the orchestrator
repository: <https://github.com/TrafficWrapper/orchestrator/blob/master/TROUBLESHOOTING.md>.
For app-specific failures, check bootstrap confirmation, signed client config,
route availability, worker health, `/tw/` distributor reachability, and APK
update signature/certificate pins.

## What Is Included

- `client/` — Android public flavor application.
- `core/` — Go transport code used to build `transport.aar`.
- `build/` — Docker-based build scripts for the AAR, APK, and update manifest.

No keystore, generated AAR, APK, or private update key is stored in this
repository.

## Requirements

- Docker, or local Go 1.24+, Android SDK, JDK 17, and gomobile.
- Your own Android release keystore.
- Your own update minisign key for update manifests.
- For running orchestrator/worker only, 1 CPU and 1 GB RAM with swap can be
  enough.
- For building the APK, plan for the Docker Android builder image, Android NDK
  and Xray artifacts: at least 15-20 GB free disk and 4 GB+ RAM with swap.

Install Docker on a fresh Linux host:

```sh
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker "$USER"
```

## Keys and Signing

TrafficWrapper uses separate trust roots for different jobs:

| Key | Who generates it | Where to store private material | Where the public part goes |
| --- | --- | --- | --- |
| Android release keystore | You, once per app lineage | Offline or CI secret storage | The certificate SHA-256 goes into `TW_PUBLIC_SIGNING_CERT_SHA256`; Android verifies APK signatures. |
| Update minisign key | You, once per update channel | Offline; never on the orchestrator if you sign manifests yourself | `update.pub` is pinned in bootstrap/orchestrator as `update_pubkey`. |
| Config-signing key | Orchestrator signer process | Orchestrator state / signer process | Public config key is pinned in bootstrap payloads. |
| Orchestrator Noise static key | Orchestrator | Orchestrator state | Public key is included in bootstrap payloads as `orch_noise_public`. |

Create an Android release keystore yourself and keep it outside Git:

```sh
keytool -genkeypair \
  -keystore /secure/path/trafficwrapper-release.jks \
  -alias trafficwrapper-release \
  -keyalg RSA -keysize 4096 -validity 10000 \
  -dname "CN=TrafficWrapper App, O=Your Organization, C=US"
```

The APK signing certificate SHA-256 from this keystore is pinned into the app
with `TW_PUBLIC_SIGNING_CERT_SHA256`. The public client accepts self-updates only
when the downloaded APK is signed by that same certificate.

Create an update minisign key yourself:

```sh
minisign -G -s update.key -p update.pub
```

Use `update.key` only to sign update manifests. Give `update.pub` to the
orchestrator/bootstrap as the update public key so the app can verify update
manifests before downloading APKs.

If you rely on orchestrator seed-on-first-run, the orchestrator generates an
update key in its own state. That is convenient for a first demo APK, but it is
not suitable for an owner-managed update channel: later APK publishing requires
manifests signed by the configured update public key. For a real deployment,
generate your own offline minisign key and set its public key in the
orchestrator from the first start.

## Build

Debug build:

```sh
./build/build-apk.sh
```

Release build requires your own signing key and expected signing certificate
hash:

```sh
export TW_RELEASE_KEYSTORE=/path/to/release.keystore
export TW_RELEASE_KEY_ALIAS=your_alias
read -r -s TW_RELEASE_STORE_PASSWORD
export TW_RELEASE_STORE_PASSWORD
read -r -s TW_RELEASE_KEY_PASSWORD
export TW_RELEASE_KEY_PASSWORD
export TW_PUBLIC_SIGNING_CERT_SHA256=<your-release-cert-sha256>
./build/build-release.sh
```

Do not commit keystores, passwords, generated AARs, APKs, or manifests.

## Build Variables

These variables are read by Gradle or the scripts in `build/`:

| Variable | What it is | Required | Where to get it / notes |
| --- | --- | --- | --- |
| `TW_APPLICATION_ID` | Android package name. | Optional | Defaults to `org.trafficwrapper.app`. |
| `TW_VERSION_CODE` | Android integer version code. | Optional | Defaults to `1001`; use monotonic values for releases. |
| `TW_VERSION_NAME` | Android version name. | Optional | Defaults to `public-1.0.0`; overrides `TW_PUBLIC_VERSION_NAME`. |
| `TW_PUBLIC_VERSION_NAME` | Legacy/public version-name fallback. | Optional | Used only when `TW_VERSION_NAME` is not set. |
| `TW_ENROLLMENT_SECRET` | Optional BuildConfig enrollment secret. | Optional | Usually empty for public platform builds. |
| `TW_RELEASE_KEYSTORE` | Path to your Android release keystore. | Required for release | Generated by you with `keytool -genkeypair`. |
| `TW_RELEASE_KEY_ALIAS` | Alias inside the release keystore. | Required for release | The `-alias` value used with `keytool`. |
| `TW_RELEASE_STORE_PASSWORD` | Keystore password. | Required for release | Store in CI secrets or enter interactively; never commit. |
| `TW_RELEASE_KEY_PASSWORD` | Key password. | Required for release | Often the same as store password for PKCS12 keystores. |
| `TW_PUBLIC_SIGNING_CERT_SHA256` | SHA-256 of the APK signing certificate pinned into the app and update verifier. | Required for release and update manifest | Run a first signing pass, then `apksigner verify --print-certs app-public-release.apk`; rebuild with the lowercase digest. This two-pass build lets the app pin its own update signing certificate. |
| `TW_ANDROID_IMAGE` | Docker image name for the Android builder. | Optional | Defaults to `trafficwrapper/android-builder:api36`. |
| `TW_UPDATE_MANIFEST_KEY` | Minisign secret key used by `build/make-manifest.sh`. | Required for manifest | Generated by `minisign -G -s update.key -p update.pub`. |
| `TW_UPDATE_OUT_DIR` | Output directory for generated update artifacts. | Optional | Defaults to `build/update-release`. |
| `TW_MIN_SUPPORTED_VERSION` | Minimum supported version in update manifest. | Optional | Defaults to `1`. |
| `TW_UPDATE_MANDATORY` | Whether update is marked mandatory. | Optional | Defaults to `false`. |
| `TW_UPDATE_NOTES` | Release notes embedded into update manifest. | Optional | Defaults to `Bug fixes and improvements.` |
| `TW_UPDATE_SEQ` | Monotonic update manifest sequence. | Optional | Defaults to APK `versionCode`; increase for every published update. |
| `TW_XRAY_VERSION` | Xray-core Android release tag. | Optional | Defaults to `v26.3.27`. |
| `TW_XRAY_ASSET` | Xray Android asset file name. | Optional | Defaults to `Xray-android-arm64-v8a.zip`. |
| `TW_XRAY_SHA256` | Expected SHA-256 of the Xray zip. | Optional | Defaults to the pinned value in `prepare-xray-android.sh`. |
| `TW_XRAY_URL` | Full Xray asset URL. | Optional | Built from `TW_XRAY_VERSION` and `TW_XRAY_ASSET` by default. |
| `CGO_LDFLAGS` | Extra CGO linker flags appended during `transport.aar` build. | Optional | Advanced native build tuning only. |
| `JAVA_OPTS`, `GRADLE_OPTS`, `DEFAULT_JVM_OPTS` | Standard Gradle wrapper JVM options. | Optional | Use only for local build tuning. |
| `GRADLE_USER_HOME`, `GOPATH`, `GOCACHE`, `GOMODCACHE` | Internal cache paths passed into the build container. | Internal | Set by scripts; normally do not override. |
| `ANDROID_CMDLINE_TOOLS_ZIP`, `ANDROID_CMDLINE_TOOLS_SHA1`, `ANDROID_NDK_VERSION`, `GOMOBILE_VERSION` | Docker build arguments in `build/android.Dockerfile`. | Optional build args | Override with `docker build --build-arg ...` only if you maintain a custom builder. |
| `ANDROID_HOME`, `ANDROID_SDK_ROOT`, `ANDROID_NDK_HOME` | Android SDK paths inside the builder image. | Internal | Set by `build/android.Dockerfile`. |

## How to Verify a Release Build

Use the minimal verifier in `build/verify-release.sh` for local release assets.
It checks the APK SHA-256, APK signing certificate SHA-256, optional minisign
manifest signature, and optional rebuild from a git tag.

```sh
APK=TrafficWrapper-app-v0.1.12.apk \
EXPECTED_APK_SHA256=adac9c6944999c871bb4e003dec9fe60597842579cf41ae93ab1e77305b112e5 \
EXPECTED_CERT_SHA256=bb8fcd34383b32c595c7d28a09cf7b89b473b86b632f3c1f5e722b4fa36e97d8 \
./build/verify-release.sh
```

If you also have an update manifest and public update key:

```sh
APK=TrafficWrapper-app-v0.1.12.apk \
MANIFEST=update-manifest.json \
MINISIG=update-manifest.json.minisig \
MINISIGN_PUBKEY=<update.pub line> \
./build/verify-release.sh
```

Byte-for-byte APK comparison after `TAG=<release-tag>` requires the same
keystore and build inputs used for that APK. Operators using their own keystore
can use the script to verify their own release channel reproducibly.

## Bootstrap

Install the APK, open the app, and import a bootstrap payload generated by the
orchestrator owner. In the orchestrator web admin, open **Devices** -> **+ New
device** and choose QR, Base64, or downloaded `.json`. The payload pins the
orchestrator Noise key and config minisign key. The app enrolls, receives
per-device credentials, verifies signed `client-config-v1`, and connects
through the configured workers.

Supported import methods:

- Tap **Scan QR** in the app and point the camera at the QR shown in the web
  admin.
- Paste the Base64 or JSON string into the bootstrap field and tap **Import**.
- Download the `.json` or `.txt` file from the web admin, open it on the phone,
  or tap **Open bootstrap file** in the app.
- Share the bootstrap text from another app to TrafficWrapper with Android
  Share.

## Security Notes

- Use your own release keystore and update minisign key.
- Keep signing keys offline or in CI secret storage.
- Public telemetry is off by default and must be enabled by the user.
- Updates are verified by minisign manifest signature, APK SHA-256, and APK
  signing certificate.
